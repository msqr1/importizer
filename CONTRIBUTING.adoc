= Contributing
:toc: preamble

Thank you for contributing to importizer!

== Internals
Although not a hard requirement, you'd want to know some internals of importizer. It's written out
link:Internals.adoc[here]. It covers architecture, design decisions, transformations &
their mechanism as well as some history & fun facts!

== Environment
=== If you're on Linux
Because your Linux is likely based on glibc, development should be done in an Alpine container.
There's https://github.com/msqr1/importizer/pkgs/container/importizer-alpine[prebuilt ones] with all
the required tools. Personally, I use https://distrobox.it[distrobox] because it makes the
host-container boundary (terminal-wise) almost invisible. Here's how to set it up:
[,bash]
----
# Create distrobox container (replace {arch} with your architecture)
distrobox create -n importizer-alpine -i ghcr.io/msqr1/importizer-alpine:3.24.0-{arch} --no-entry

# Enter container
distrobox enter importizer-alpine
----

=== Setup
Setup the environment (check architecture, required tools, backend & set environment variables) by
running (replace {os} with the correct file):
[,bash]
----
. scripts/setup-env/{os}
----

Note:

* All scripts currently check if CMake and Ninja is in PATH.
* *Linux:* The script checks if Clang and LLD is in PATH.
* *MacOS:* The script checks Homebrew LLVM (for Clang and LLD).
* *Windows:* There's 2 optional arguments:
[%autowidth]
|===
|Argument| Description| Default value

|llvmPrefix
|Set the directory of the LLVM installation
|`C:/Program Files/LLVM`

|cmakePrefix
|Set the directory of the CMake installation
|`C:/Program Files/CMake`
|===
They can take the special value of `:bundled` that tells the script to look for the installation
bundled within Visual Studio or Build Tools. Note that bundled tools usually lag behind in version,
so use `:bundled` with cautions.

== Dependency linkage
[%autowidth]
|===
| |Linux |MacOS |Windows

|3rd-party libraries
3+^.^|Static

|C++ Standard Library (CxxStdlib)
|Static
.2+^.^|Dynamic^1^
|Static

|C Standard Library (CStdlib)
|Static^2^
|Dynamic^3^
|===

. CStdlib is libSystem, CxxStdlib is a modded libc{plus}{plus}, both are dynamic but with stable ABI
between versions guaranteed by MacOS so we can treat as if they're static.
. CStdlib is Windows UCRT, it's dynamic but also with stable ABI guaranteed between versions by
Windows so we can use as if it's static.
. CStdlib is musl libc. glibc was considered, but:
.. Static linking with it is unsupported.
.. Even if we do dynamic linking, it's still only on most systems, not all.
.. There's no ABI stability guarantee between versions.

== Building
=== Skip building LibTooling
Importizer needs Clang LibTooling. But LibTooling isn't like other 3rd-party libraries, it is
massive! To save one from compiling it, there's
https://github.com/msqr1/importizer/releases/tag/libtooling-22.1.6[prebuilt ones]. Since C++ ABIs &
LLVM bitcode formats can change between versions, if you match the toolchain in the table below that
was used to build LibTooling for CI, the prebuilt LibTooling will integrate seamlessly. One can
start coding right away.
[,bash]
----
cmake -P scripts/libtooling/Ensure.cmake
----
[%autowidth]
|===
| |Linux |MacOS |Windows

|Build system
3+^.^|CMake 3.31{plus}

|Generator
3+^.^|Ninja^1^

|Compiler toolchain
|clang, ld.lld 22.1.3
|clang 20.1.8 (Homebrew)
|clang-cl, lld-link, llvm-ml 20.1.x^2^

|Backend
|Alpine 3.24.0
|Xcode SDK (`SDKROOT`)
|Visual Studio or Build Tools^3^
|===

. Ninja is the only generator that meets all these requirements:
.. Available on all supported OS'es & architectures
.. Can generate JSON compilation database
. `20.1.8` on AMD64, `20.1.6` on ARM64. Basically replacing MSVC toolchain here.
. Used strictly to provide the Windows SDK.

Once you installed the toolchain, create a `CMakeUserPresets.json` and inherit from
`CMakePresets.json` to customize your local build. This is so you don't have to derive the correct
settings yourself. For example, on MacOS:
[,json]
----
{
  "version": 10,
  "configurePresets": [
    {
      "name": "mine",
      "inherits": ["importizer-macos"],

      "$comment": "Easily switch between debug & release configuration",
      "generator": "Ninja Multi-Config"
    }
  ]
}
----

=== Building from source entirely
You are free to use other compiler toolchains (GCC, MSVC, Apple Clang, another Clang), but you will
have to rebuild LibTooling with that toolchain. Good luck! Note that everything else here assumes
you won't be doing this.

=== Build process
Ok, enough talking, here's how to build the release mode from scratch. Note that cross compilation
is currently unsupported.
[,bash]
----
# Shallow clone to save space
git clone --depth 1 https://github.com/msqr1/importizer
cd importizer

# Setup the environment
. scripts/setup-env/{OS}

# Ensure LibTooling
cmake -P scripts/libtooling/Ensure.cmake

# Configure
cmake -B build -DCMAKE_BUILD_TYPE=Release

# Build using all cores
cmake --build build --config Release -j $(cmake -P scripts/Nproc.cmake)
----
The binary is in `build`, named `importizer` on Linux or MacOS and `importizer.exe` on Windows.

== Testing
=== Running tests
Tests are disabled by default, but can be toggled via the `BUILD_TESTING` CMake option.
[,bash]
----
# Configure with tests enabled (Debug mode recommended for testing)
cmake -B build -DCMAKE_BUILD_TYPE=Debug -DBUILD_TESTING=ON

# Build using all cores
cmake --build build --config Debug -j $(cmake -P scripts/Nproc.cmake)

# Run tests with recommended settings & using all cores
ctest --test-dir build/tests \
  --output-on-failure \
  --progress \
  --schedule-random \
  -j $(cmake -P scripts/Nproc.cmake)
----

=== Adding new tests
Importizer uses a custom directory-based test harness (located in `src/run-test`). To add a new test
case:

. Create a new directory under `tests/` with a descriptive name (e.g., `tests/my-new-feature/`).
. Add a `Config.toml` file inside this directory to configure the importizer run.
. Add a `RefCli.txt` file containing the expected CLI output.
. *(Optional)* Create a `in/` subdirectory containing the input files (C{plus}{plus} header-based
code).
. *(Optional)* Create a `ref/` subdirectory containing the expected output files. The test driver
will automatically compare the actual output directory against this reference directory. Don’t add
this when expecting an error.

The CMake script dynamically globs the `tests/` folder, so your new test will be automatically
picked up on the next `cmake -B build` run.

== Coding conventions
* 100-column per line limit for files & CLI outputs

=== Naming
* *All code variables & functions:* `camelCase`
* *C{plus}{plus} classes, types, & files:* `PascalCase` (e.g., `MyClass.cc`) unless it's an
abbreviation
* *Executables & directories:* `kebab-case` (e.g., `run-test/`)
* *Environment variables:* `ALL_CAPS_SNAKE_CASE` prefixed with `IMPORTIZER_` (e.g., `IMPORTIZER_OS`)

=== C{plus}{plus} style
* Follow the LLVM coding style.
* Use `llvm::StringRef` (not `std::string_view` because it's missing many features like `split`)
over `const char*`, & `std::array`/`std::vector` over C-style arrays.
* Use angled brackets (`#include <header.h>`) for external/third-party headers.
* The C{plus}{plus} file containing `main()` for any executable must be named `Driver.cc`.
* Maintain the exact same ordering of options in the Options section & parsing logic in `getOpts`.
* Do not use RTTI, if you must, use https://llvm.org/docs/HowToSetUpLLVMStyleRTTI.html[LLVM's]

===  C{plus}{plus} error handling
* *Do not* use C{plus}{plus} Exceptions
* *Return Values:* Mark fallible functions with `[nodiscard]` & return `bool` (or a type
implicitly convertible to `bool`).
* *Logging:* Functions must handle logging their own errors & should be marked `noexcept`.
* *System Errors:* Mark branches handling system-level errors (e.g., `fseek` or `ftell` failures)
with `[unlikely]`. Do not use this for input validation.

=== CMake
* Always place `common-config` as the *first* item in your `target_link_libraries` list.
* Only use generator expressions for build-time conditionals (e.g., `<$COMPILE_LANGUAGE:CXX>`).
* Check before adding a compiler/linker flag if it's not guaranteed to be available on supported
compiler (e.g., `-fsanitize=address`).
* `3rd-party` directory must only contain 3rd-party libraries used by the project
* Use `target\_*_*` to add flags for this project, & `add\_*_*` to add for this & all deps.
* If a global setting affects LibTooling too (e.g. `BUILD_SHARED_LIBS`), put in `CMakePresets.json`.
Otherwise (eg. `CMAKE_CXX_SCAN_FOR_MODULES`), put in `CMakeLists.txt`

=== Scripts
* Use CMake to write scripts, if not possible, then make a folder & add scripts for all supported
OS'es (like `setup-env`).
* Environment setup scripts define these environment variables so you can use in other places:
** `IMPORTIZER_OS`: The operating system (`windows`, `linux`, or `macos`)
** `IMPORTIZER_ARCH`: The architecture (`amd64` or `arm64`)
* For CI detection check if the environment variable `CI` was defined

== Checklists
=== New release
* [ ] Update the version string(s) in:
** `.github/workflows/Release.yml`
** `ChangeLog.adoc` (delete "in progress")
** `src/importizer/Opts.cc`
** `CMakeLists.txt`
* [ ] Commit & run the Tag Release workflow, it will:
** Check if the last commit built & tested successfully (in Debug mode).
** Check if LibTooling binaries are
** Build & test on its own (in Release mode).
** Tag the new release.

=== Bumping LibTooling version
* [ ] Tag a new release with (replace `{v}` with the new version):
** Title: `libtooling-{v}`
** Tag: `libtooling-{v}`
** Description: `LibTooling {v} for developers & CI.`
** Pre-release
* [ ] Update the version string(s) in:
** `scripts/libtooling/Ensure.cmake` (also update LLVM Project source hash)
** `CONTRIBUTING.adoc`
* [ ] Add `Bump LibTooling to {v}` to `ChangeLog.adoc`
* [ ] Commit with message `Bump LibTooling to {v}` & let the CI run & wait for completion
* [ ] For the next commit:
** Update the LibTooling hashes in `scripts/libtooling/Ensure.cmake`

=== Bumping Alpine images version
* [ ] Update the version string(s) in:
** `.github/workflows/Alpine.yml`
** `scripts/alpine/Containerfile`
** `CiMatrix.json`
** `CONTRIBUTING.adoc`
* [ ] Add `Bump Alpine to {v}` to `ChangeLog.adoc`
* [ ] Commit with message `[no ci] Bump Alpine to {v}`
* [ ] Run the Make Alpine Images workflow & wait for completion
* [ ] Run the Tag Continuous workflow

=== Permissions
When pushing, ensure permission of files in `scripts/` is set to executable. You can use the
provided git hook to do this for you:
[,bash]
----
git config core.hooksPath .githooks
----
